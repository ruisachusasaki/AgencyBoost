CONTEXT:
This is AgencyBoost, a CRM for marketing agencies. We are building a 
NEW HIRE WORKFLOW feature. The following has already been completed:
  - DB tables: ic_agreement_templates, job_offers, offer_signatures, 
    offer_status_log
  - New status values added to the applicants table: 'send_offer', 
    'offer_accepted', 'offer_declined', 'hired'

This prompt builds the IC AGREEMENT TEMPLATE BUILDER — where admins 
create and manage the Independent Contractor Agreement template that 
gets dynamically populated when an offer is sent to an applicant.

This lives under Settings > HR Settings as a new tab called 
"IC Agreement".

ALWAYS use the primary teal theme color (hsl(179, 100%, 39%) / #00C9C6) 
for buttons, accents, and interactive elements. Never use blue.

═══════════════════════════════════════
PART A — BACKEND API ROUTES
═══════════════════════════════════════

Add the following routes to server/routes.ts. All require 
requireAuth(). Admin only — Managers and regular Users cannot 
access these endpoints.

─────────────────────────────────────
GET /api/ic-agreement-templates
─────────────────────────────────────
- Return all IC Agreement templates ordered by createdAt DESC
- Include: id, name, isActive, createdBy (with staff name joined), 
  createdAt, updatedAt
- Do NOT return the full content field in the list — keep it 
  lightweight

─────────────────────────────────────
GET /api/ic-agreement-templates/:id
─────────────────────────────────────
- Return full template including content field
- Include createdBy staff name

─────────────────────────────────────
GET /api/ic-agreement-templates/active
─────────────────────────────────────
- Return the single active template (isActive = true)
- Include full content field
- If no active template exists return { template: null }
- This endpoint will be used by the Send Offer flow in Prompt C

─────────────────────────────────────
POST /api/ic-agreement-templates
─────────────────────────────────────
- Admin only
- Body: { name, content, isActive }
- Set createdBy = req.user.id
- If isActive = true: first set ALL other templates 
  isActive = false in the same transaction, then insert 
  the new template with isActive = true
- Return created template

─────────────────────────────────────
PUT /api/ic-agreement-templates/:id
─────────────────────────────────────
- Admin only
- Body: { name, content, isActive } — all optional
- If isActive = true is being set: first set ALL other 
  templates isActive = false in the same transaction
- Update updatedAt = now()
- Return updated template

─────────────────────────────────────
DELETE /api/ic-agreement-templates/:id
─────────────────────────────────────
- Admin only
- Soft check: if this template is referenced by any job_offers 
  (templateId = this id), do NOT delete — return 400 with message 
  "Cannot delete — this template has been used in sent offers."
- If the template being deleted is currently isActive, return 400 
  with message "Cannot delete the active template. Set another 
  template as active first."
- Otherwise hard delete

ICAgreementEditor.tsx (Create/Edit Page)
─────────────────────────────────────
Full-page editor for creating or editing an IC Agreement template.

This page is used for both NEW (/settings/hr/ic-agreement/new) 
and EDIT (/settings/hr/ic-agreement/:id) flows.

Layout:

LEFT PANEL (2/3 width) — Editor:

  - Rich text editor for the agreement content

RIGHT PANEL (1/3 width) — Placeholder Toolbar + Settings:

    Label: "Make this the active template"
    Subtext: "Only one template can be active at a time.

    checked and disabled (cannot deactivate without 
    activating another)

BOTTOM ACTION BAR (full width, sticky):
  - Left: "Back to Templates" link

  - Both buttons show loading state during save

- Do NOT modify any existing HR Settings tabs or their content
- Do NOT modify any existing routes — only ADD new ones
- Do NOT touch vite.config.ts, package.json, or drizzle.config.ts
- Do NOT modify any existing schema or migrations
- Do NOT introduce a new rich text editor library — use whatever

- Do NOT modify any existing applicant tracking components

  useQuery({ queryKey: [...], queryFn: ... })

- Toast notifications: only "default" and "destructive" variants
- Icons from lucide-react only

  enforce this in BOTH the API (transaction) and the UI 
  (confirmation dialogs)

  be atomic — use a database transaction
