CONTEXT

This is Step 9: Assignment Rules Integration. We want tasks to automatically route to the correct person based on form answers. Assignment should be based on ROLE/POSITION, not specific users, so if the Creative Project Manager changes, we just update who holds that role.

The task table has an "assignedTo" field for the assigned user, and a "categoryId" field for task category. We have existing categories for Creative, DevOps, and Data.

WHAT TO BUILD

1. Assignment Rules Table

Create a new table to store assignment rules:

task_intake_assignment_rules
- id: uuid, primary key
- formId: uuid, foreign key to task_intake_forms
- ruleName: text (admin-friendly name like "Creative Tasks to Creative PM")
- conditions: jsonb (same structure as section visibility conditions)
- assignToRole: text (role/position name like "Creative Project Manager")
- assignToUserId: uuid, nullable (optional direct user assignment as fallback)
- setCategoryId: uuid, nullable, foreign key to task categories (auto-set category)
- setTags: text array, nullable (auto-add these tags)
- priority: integer (lower number = higher priority, for rule ordering)
- isActive: boolean, default true
- createdAt: timestamp
- updatedAt: timestamp

2. Role-to-User Mapping Table

Create a role_assignments table:

role_assignments
- id: uuid, primary key
- roleName: text (e.g., "Creative Project Manager", "DevOps Lead", "Data Team Lead")
- userId: uuid, foreign key to users
- isActive: boolean, default true
- createdAt: timestamp
- updatedAt: timestamp

This allows roles to be mapped to users. When someone changes positions, just update the role assignment.

5. Assignment Rule Evaluation Function

Create a function: evaluateAssignmentRules(rules, answers)

Returns: { assignToUserId, categoryId, tags }

Logic:
1. Sort rules by priority ascending (lower number = higher priority)
2. For each rule where isActive = true:
   a. Evaluate conditions against answers using same logic as section visibility
   b. If all conditions match:
      - Look up assignToRole in role_assignments to get userId
      - If role has no userId assigned, use rule's assignToUserId as fallback
      - Return { assignToUserId: userId, categoryId: rule.setCategoryId, tags: rule.setTags }
3. If no rules match, return { assignToUserId: null, categoryId: null, tags: [] }

6. Update Task Creation Flow

Modify POST /api/task-intake/submit:

After generating description and before creating task:
1. Fetch all active assignment rules for this form, ordered by priority
2. Call evaluateAssignmentRules(rules, answers)
3. When creating the task, use:
   - assignedTo: result.assignToUserId (if not null)
   - categoryId: result.categoryId (if not null)
   - tags: merge result.tags with any existing tags from the task

RULES LIST as a table:

Each row shows:

11. API Endpoints for Assignment Rules

GET /api/task-intake/assignment-rules
- Returns all rules ordered by priority
- Include human-readable condition summary
- Include current user name for the role

POST /api/task-intake/assignment-rules
- Create new rule
- Body: { ruleName, conditions, assignToRole, setCategoryId, setTags, priority, isActive }

PUT /api/task-intake/assignment-rules/:id
- Update existing rule

DELETE /api/task-intake/assignment-rules/:id
- Delete rule

12. API Endpoints for Roles

- Returns all roles with current assigned user info

POST /api/assignment-roles
- Create new role
- Body: { roleName, userId }

PUT /api/assignment-roles/:id
- Update role (change name or assigned user)
- Body: { roleName, userId }

DELETE /api/assignment-roles/:id
- Delete role (should warn if rules reference this role)

13. Handle Edge Cases

No matching rule:
- Task is created with assignedTo = null
- Task appears in unassigned queue
- This is fine - allows manual assignment for edge cases

Role has no assigned user:
- Check rule's assignToUserId as fallback
- If still null, task created without assignment
- Show visual indicator in Roles list that role needs user assigned

Multiple rules could match:
- First matching rule by priority wins
- If same priority, first by creation date wins

Deleting a role that's used in rules:
- Show warning: "This role is used in X assignment rules"
- Either prevent deletion or offer to update rules
