CONTEXT

This is Step 7: Task Creation & Answer Storage. When a user submits the intake form, we need to create a task and store all their answers.

WHAT TO BUILD

1. Answer Storage Table

Create a new table to store form submissions:

task_intake_submissions
- id: uuid, primary key
- taskId: uuid, foreign key to tasks table (nullable initially, linked after task created)
- formId: uuid, foreign key to task_intake_forms
- submittedBy: uuid, foreign key to users table
- submittedAt: timestamp
- status: text, default 'pending' (values: pending, task_created, cancelled)
- createdAt: timestamp
- updatedAt: timestamp

task_intake_answers
- id: uuid, primary key
- submissionId: uuid, foreign key to task_intake_submissions
- questionId: uuid, foreign key to task_intake_questions
- sectionId: uuid, foreign key to task_intake_sections
- answerValue: text (store all answers as text/JSON string)
- wasVisible: boolean (true if the section was visible when submitted)
- createdAt: timestamp

2. Submission API Endpoint

POST /api/task-intake/submit

Request body:
{
  "formId": "uuid",
  "answers": {
    "questionId1": "answer value",
    "questionId2": ["multi", "choice", "values"],
    "questionId3": "2025-01-15"
  },
  "visibleSectionIds": ["uuid1", "uuid2", "uuid3"]
}

This endpoint should:

Step A - Validate submission:
- Verify all required questions in visible sections have answers
- Verify answer formats are correct (dates, numbers, etc.)
- Return 400 with errors if validation fails

Step B - Create submission record:
- Create task_intake_submissions record with status 'pending'
- Create task_intake_answers records for each answer
- Set wasVisible = true for answers in visibleSectionIds, false otherwise

Step C - Extract key fields for task:
- Find answer for "task_name" (internalLabel) → task title
- Find answer for "client_select" → client association
- Find answer for "due_date" → task due date
- Find answer for "priority_level" → task priority
- Find answer for "department" → task department/category

Step D - Create the task:
- Create record in your existing tasks table
- Use extracted fields from Step C
- Link submissionId to the task
- Set task status to your default initial status

Step E - Update submission:
- Update submission record with taskId
- Set status to 'task_created'

Step F - Return response:
{
  "success": true,
  "submissionId": "uuid",
  "taskId": "uuid",
  "taskUrl": "/tasks/uuid"
}

3. Answer Retrieval Endpoint

GET /api/task-intake/submissions/:submissionId

Returns:
{
  "id": "uuid",
  "taskId": "uuid",
  "submittedBy": { "id": "uuid", "name": "User Name" },
  "submittedAt": "2025-01-23T...",
  "status": "task_created",
  "sections": [
    {
      "sectionId": "uuid",
      "sectionName": "Task Basics",
      "wasVisible": true,
      "answers": [
        {
          "questionId": "uuid",
          "questionText": "What is the name of this task?",
          "internalLabel": "task_name",
          "answerValue": "New video ad for Tax Lien Code",
          "questionType": "text"
        }
      ]
    }
  ]
}

This endpoint structures answers by section for easy display.

4. Get Submission by Task ID

GET /api/tasks/:taskId/intake-submission

Returns the same structure as above, but looks up by taskId instead of submissionId.
This lets you view the original intake answers from the task detail page.
