PART A — BACKEND API ROUTES
═══════════════════════════════════════

Add the following routes to server/routes.ts. All require requireAuth().
All require Admin or Manager role — regular Users cannot access these.

─────────────────────────────────────
GET /api/onboarding/instances
─────────────────────────────────────
- Admins: return ALL onboarding instances across all teams
- Managers: return only instances where the staff member's teamId 
  matches the manager's assigned team
- Query params (all optional):
    ?status=active|completed|all  (default: active)
    ?teamId=123                   (admin only filter)
    ?search=name                  (filter by staff member name)
- Join and return for each instance:
    - Staff member: id, firstName, lastName, email, position, 
      teamId, teamName, avatarUrl
    - Instance: id, status, startDate, completedAt, createdAt,
      totalDays (from templateSnapshot->totalDays),
      dayUnlockMode (from templateSnapshot->dayUnlockMode)
    - Progress stats (compute in SQL or application layer):
        totalItems: count of onboarding_instance_items for instance
        completedItems: count where isCompleted = true
        requiredItems: count where isRequired = true
        completedRequiredItems: count where isRequired = true 
          AND isCompleted = true
        currentDay: days elapsed since startDate (min 1, max totalDays)
        percentComplete: (completedRequiredItems / requiredItems) * 100
        isBehind: boolean — true if there exist isRequired = false AND 
          isCompleted = false items from days BEFORE currentDay
- Order by: startDate DESC

─────────────────────────────────────
GET /api/onboarding/instances/:instanceId
─────────────────────────────────────
- Return full instance detail for a single onboarding instance
- Include all fields from the list endpoint above
- Additionally include ALL onboarding_instance_items ordered by 
  dayNumber ASC, sortOrder ASC
- For each item: attempt to join KB article title and training 
  course title via referenceId (same as Prompt 4 Part A)
- Managers can only access instances belonging to their team
- Admins can access any instance

─────────────────────────────────────
PATCH /api/onboarding/instances/:instanceId/items/:itemId
─────────────────────────────────────
- Allows a Manager or Admin to check/uncheck items on behalf 
  of the hire
- Body: { isCompleted: boolean }
- Same completion logic as the hire's own PATCH endpoint from 
  Prompt 4:
    - Mark complete: isCompleted = true, completedAt = now(), 
      completedBy = req.user.id
    - Uncheck: isCompleted = false, completedAt = null, 
      completedBy = null, autoCompleted = false
- After update: check if ALL isRequired items are complete — 
  if yes, mark instance status = 'completed', completedAt = now()
- Return updated item

─────────────────────────────────────
PATCH /api/onboarding/instances/:instanceId/status
─────────────────────────────────────
- Admin or Manager only
- Body: { status: 'active' | 'paused' | 'completed' }
- Updates instance status
- Used for manually pausing or reopening an onboarding instance
- Return updated instance

OnboardingDashboard.tsx (Main Page)
─────────────────────────────────────
Page header: "Onboarding Dashboard" with subtitle "Track your 
team's new hire progress"

TOP SUMMARY STRIP — 4 stat cards in a row:
  1. Active Onboardings — count of active instances
  2. Completed This Month — count completed in current calendar month
  3. Behind Schedule — count where isBehind = true
  4. Avg. Completion — average percentComplete across active instances
  All stat cards use teal accent for the number value.

FILTERS BAR (below summary strip):
  - Search input: filter by hire name (debounced 300ms)
  - Status filter: dropdown — "Active", "Completed", "All"
  - Department filter (Admin only): dropdown populated from teams API
  - All filters update the query params and refetch

MAIN TABLE — one row per onboarding instance:

Columns:
  1. HIRE
     - Avatar + full name (bold) + position title beneath in gray
     
  2. DEPARTMENT
     - Team/department name
     
  3. START DATE
     - Formatted date (e.g. "Mar 10, 2026")
     
  4. PROGRESS
     - OnboardingProgressBar component (teal fill)
     - "X of Y required items · Z%" beneath bar in gray small text
     
  5. DAY
     - "Day X of N" 
     - If currentDay > totalDays: show "Day N of N (Complete)"
     
  6. STATUS BADGE
     - Active: teal badge
     - Completed: green badge
     - Paused: gray badge
     - Behind: red badge (overrides Active if isBehind = true)
     
  7. ACTIONS
     - "View Details" button (outline, teal) — opens 
       OnboardingInstanceDrawer
     - Kebab menu (three dots) with options:
         "Pause Onboarding" (if active)
         "Resume Onboarding" (if paused)
         "Mark as Complete"
       Each calls PATCH /api/onboarding/instances/:id/status with 
       appropriate status value and confirmation dialog before action

Empty state (no instances):
  - Icon: Users from lucide-react
  - "No active onboardings"
  - "New hire onboarding checklists will appear here when staff 
    members are added with a matching position template."

OnboardingInstanceDrawer.tsx
─────────────────────────────────────
A slide-over drawer (from the right) that opens when a manager 
clicks "View Details" on a hire's row.

Query GET /api/onboarding/instances/:instanceId on open.

DRAWER HEADER:
  - Avatar + hire's full name + position
  - Status badge
  - Start date + "Day X of N"
  - Progress bar (OnboardingProgressBar, md size)
  - "X of Y required items complete"
  - Close button (X, top right)

DRAWER BODY — full checklist grouped by day:
  Same day-by-day accordion layout as the hire's own view 
  (ChecklistDaySection style) but in a condensed read-for-manager 
  format:

  Each day section:
  - "Day N — [date]" header
  - Day completion badge if all required items done
  - Items listed beneath with:
      - Circular checkbox (manager CAN toggle these — calls 
        PATCH /api/onboarding/instances/:instanceId/items/:itemId)
      - Title + type badge
      - If isCompleted: "Completed [date] by [name]" in gray small text
      - If autoCompleted: teal "Auto-completed via Training" badge
      - If isBehind item (required, incomplete, day < currentDay): 
          red "Overdue" badge

  Locked days: still shown but labeled "🔒 Not yet unlocked" — 
  manager CAN still check items in locked days (manager override)

DRAWER FOOTER:
  - "Pause Onboarding" / "Resume" button (if applicable)
  - "Mark as Complete" button
  Both with confirmation dialogs before action.

After any item toggle or status change in the drawer:
  - Invalidate /api/onboarding/instances query so the table 
    updates automatically in the background
