CONTEXT

This is Step 9: Assignment Rules Integration. We want tasks to automatically route to the correct person based on form answers. Assignment should be based on ROLE/POSITION, not specific users, so if the Creative Project Manager changes, we just update who holds that role.

The task table has an "assignedTo" field for the assigned user, and a "categoryId" field for task category. We also have existing categories for Creative, DevOps, and Data.

WHAT TO BUILD

1. Assignment Rules Table

Create a new table to store assignment rules:

task_intake_assignment_rules
- id: uuid, primary key
- formId: uuid, foreign key to task_intake_forms
- ruleName: text (admin-friendly name like "Creative Tasks to Creative PM")
- conditions: jsonb (same structure as section visibility conditions)
- assignToRole: text (role/position name like "Creative Project Manager")
- assignToUserId: uuid, nullable (optional direct user assignment as fallback)
- setCategoryId: uuid, nullable, foreign key to task categories (auto-set category)
- setTags: text array, nullable (auto-add these tags)
- priority: integer (lower number = higher priority, for rule ordering)
- isActive: boolean, default true
- createdAt: timestamp
- updatedAt: timestamp

2. Role-to-User Mapping

We need a way to map roles to current users. Options:

Option A: Add a "role" field to your existing users/staff table if not present
Option B: Create a simple role_assignments table:

role_assignments
- id: uuid, primary key
- roleName: text (e.g., "Creative Project Manager", "DevOps Lead", "Data Analyst Lead")
- userId: uuid, foreign key to users
- isActive: boolean, default true
- createdAt: timestamp
- updatedAt: timestamp

This allows one role to be held by one person (or we could allow multiple and pick the first active one).

Check if your users table already has a role or position field. If yes, we can use that. If not, create the role_assignments table.

4. Assignment Rule Evaluation Function

Create a function:

evaluateAssignmentRules(rules, answers) returns { assignToUserId, categoryId, tags }

Logic:
1. Sort rules by priority (ascending - lower number first)
2. For each rule:
   a. Evaluate conditions against answers (same logic as section visibility)
   b. If conditions match:
      - Look up assignToRole in role_assignments to get userId
      - Return { assignToUserId: userId, categoryId: rule.setCategoryId, tags: rule.setTags }
3. If no rules match, return { assignToUserId: null, categoryId: null, tags: [] }

5. Update Task Creation Flow

Modify POST /api/task-intake/submit to use assignment rules:

After generating description and before creating task:
1. Fetch all active assignment rules for this form
2. Call evaluateAssignmentRules(rules, answers)
3. Use returned values when creating task:
   - assignedTo: result.assignToUserId (if not null)
   - categoryId: result.categoryId (if not null, otherwise keep any existing logic)
   - tags: merge result.tags with any existing tags

9. API Endpoints

GET /api/task-intake/assignment-rules
- Returns all rules ordered by priority
- Include human-readable condition summary

- Delete rule (soft delete or hard delete)

- Returns all roles with current assigned user

POST /api/assignment-roles
- Create new role

PUT /api/assignment-roles/:id
- Update role (change name or assigned user)

DELETE /api/assignment-roles/:id
- Delete role

10. Handle Edge Cases

No matching rule:
- Task is created but assignedTo remains null
- Could show warning to admin or default to a "catch-all" assignment

Role has no assigned user:
- Log warning
- Task is created with assignedTo = null
- Consider showing this in the admin UI (role needs assignment)

Multiple rules match:
- First matching rule by priority wins
- If same priority, first one in database order wins
