CONTEXT:
This is AgencyBoost, a CRM for marketing agencies. We are building a 
NEW HIRE WORKFLOW feature. The following has already been completed:
  - DB tables: ic_agreement_templates, job_offers, offer_signatures, 
    offer_status_log
  - New status values on applicants table: 'send_offer', 
    'offer_accepted', 'offer_declined', 'hired'
  - IC Agreement template builder under Settings > HR Settings > 
    IC Agreement
  - Send Offer modal + offer creation + auto-email on the applicant 
    detail page

This prompt builds the PUBLIC SIGNING PAGE — the applicant-facing 
page at /sign-offer/{token} where they read their populated IC 
Agreement, sign it by drawing or typing, and submit. No login 
required. On submission the status auto-updates and the signed 
document is permanently archived.

ALWAYS use the primary teal theme color (hsl(179, 100%, 39%) / #00C9C6) 
for buttons, accents, and interactive elements. Never use blue.

PART A — BACKEND API ROUTES
═══════════════════════════════════════

These routes are PUBLIC — they do NOT use requireAuth() because 
the applicant has no AgencyBoost account. Security is provided 
by the unique unguessable signingToken (UUID).

─────────────────────────────────────
GET /api/sign-offer/:token
─────────────────────────────────────
Purpose: Load the offer details for the signing page.

Logic:
  - Query job_offers where signingToken = token
  - If not found: return 404 with { error: 'not_found' }
  - If offer.status = 'signed': return 200 with 
    { status: 'already_signed', signedAt: offer signature 
    timestamp }
  - If offer.status = 'declined': return 200 with 
    { status: 'already_declined' }
  - If offer.expiresAt is set AND now() > expiresAt: 
    return 200 with { status: 'expired' }
  - Otherwise return:
    {
      status: 'pending',
      offer: {
        id,
        populatedContent,   — the full HTML agreement
        compensation,
        compensationType,
        startDate,
        sentAt
      },
      applicant: {
        name,               — from joined applications table
        email,              — from joined applications table
        position
      }
    }
  - Do NOT expose: signingToken, createdBy, templateId, 
    applicationId, or any internal IDs in this response

─────────────────────────────────────
POST /api/sign-offer/:token/sign
─────────────────────────────────────
Purpose: Submit the applicant's signature and mark offer accepted.

Body:
  {
    signatureType: 'drawn' | 'typed',
    signatureData: string,  — base64 PNG data URL if drawn, 
                              plain text name if typed
    signerName: string,     — full name entered by applicant
    signerEmail: string     — email confirmed by applicant
  }

Logic:

STEP 1 — Validate token and offer state
  - Query job_offers where signingToken = token
  - If not found: return 404
  - If offer.status != 'pending': return 400 with message 
    "This offer has already been signed or declined."
  - If expiresAt is set and expired: return 400 with message 
    "This signing link has expired. Please contact your 
    hiring manager."

STEP 2 — Validate body
  - signerName: required, non-empty string
  - signerEmail: required, valid email format
  - signatureType: must be 'drawn' or 'typed'
  - signatureData: required, non-empty
  - If drawn: signatureData must start with 'data:image/png;base64,'
  - Return 400 with descriptive message if any validation fails

STEP 3 — Capture metadata
  - ipAddress: req.ip or req.headers['x-forwarded-for'] 
    (check which is available — use x-forwarded-for first 
    as the app likely sits behind a proxy)
  - userAgent: req.headers['user-agent']

STEP 4 — Save signature and update offer (in a transaction):
  - Insert into offer_signatures:
      offerId, signatureType, signatureData, signerName,
      signerEmail, ipAddress, userAgent, signedAt = now()
  - Update job_offers: status = 'signed'
  - Insert into offer_status_log:
      offerId, status = 'signed', changedBy = null,
      note = 'Applicant signed the IC Agreement'

STEP 5 — Update application status
  - Find the application via offer.applicationId
  - Update application status = 'offer_accepted'

STEP 6 — Notify the hiring manager
  - Find the job_offer.createdBy (the manager who sent the offer)
  - Use the existing notification system (same pattern as 
    onboarding notifications)
  - Notification:
      title: "[Candidate Name] signed their IC Agreement"
      message: "[Full Name] has reviewed and signed their 
      IC Agreement for the [position] role. Their start 
      date is [formatted start date]."
      userId: offer.createdBy
  - Fire-and-forget — do not await, do not block the response

STEP 7 — Return success
  - Return { success: true, message: 'Agreement signed 
    successfully.' }

─────────────────────────────────────
POST /api/sign-offer/:token/decline
─────────────────────────────────────
Purpose: Applicant declines the offer from the signing page.

Body:
  {
    reason: string (optional)
  }

Logic:

STEP 1 — Validate token
  - Query job_offers where signingToken = token
  - If not found: return 404
  - If offer.status != 'pending': return 400 "This offer 
    has already been responded to."

STEP 2 — Update offer and application (in a transaction):
  - Update job_offers: status = 'declined'
  - Insert into offer_status_log:
      offerId, status = 'declined', changedBy = null,
      note = reason if provided, else 'Applicant declined 
      the offer'
  - Update application status = 'offer_declined'

STEP 3 — Notify the hiring manager
  - Same notification pattern as above:
      title: "[Candidate Name] declined their offer"
      message: "[Full Name] has declined the offer for 
      the [position] role.[reason text if provided: 
      ' Reason: {reason}']"
      userId: offer.createdBy
  - Fire-and-forget

STEP 4 — Return { success: true }
