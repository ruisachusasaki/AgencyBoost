PART C — BACKGROUND NOTIFICATION JOBS
═══════════════════════════════════════

Create a new file: server/services/onboardingNotificationService.ts

Export two async functions following the patterns found in Part A.

─────────────────────────────────────
Function 1: runDayUnlockNotifications()
─────────────────────────────────────
Purpose: Each morning, notify new hires when a new onboarding 
day has just become available.

Logic:

STEP 1 — Check setting
  - Read onboarding_day_unlock_notifications setting
  - If false: return early

STEP 2 — Find all active onboarding instances
  - Query onboarding_instances where status = 'active'
  - Only process instances where dayUnlockMode = 'calendar'
    (completion-gated instances unlock based on actions, 
    not time — no morning notification needed)

STEP 3 — For each instance, calculate today's unlocked day
  - currentDay = business days elapsed since startDate + 1
    (skip Saturday and Sunday when counting)
  - Cap at totalDays (from templateSnapshot)
  - If currentDay = 1 AND instance was created today: skip 
    (spawn notification already sent at creation — avoid 
    duplicate on Day 1)

STEP 4 — Check if this day was JUST unlocked today
  - We only want to notify on the FIRST run of each new day
  - Check: does a notification already exist for this staffId 
    with title matching "Day {currentDay} of your onboarding 
    is ready" created today (same calendar date)?
  - If yes: skip (already notified today)
  - If no: proceed

STEP 5 — Check there are actually items in this day
  - Query onboarding_instance_items where instanceId = id 
    AND dayNumber = currentDay
  - If no items exist for this day: skip

STEP 6 — Send notification to the hire
  - title: "Day {currentDay} of your onboarding is ready"
  - message: "Your Day {currentDay} onboarding items are now 
    unlocked. Head to HR > Onboarding Checklist to continue."
  - userId: instance.staffId

  Wrap all per-instance logic in try/catch — one failing 
  instance must not stop others from processing.

─────────────────────────────────────
Function 2: runBehindScheduleAlerts()
─────────────────────────────────────
Purpose: Alert managers when a new hire has fallen behind on 
their onboarding checklist.

Logic:

STEP 1 — Read threshold setting
  - Read onboarding_behind_schedule_threshold (default 2)
  - threshold = number of days

STEP 2 — Find all active instances that are behind
  - Query onboarding_instances where status = 'active'
  - For each instance, compute currentDay (same business day 
    logic as above)
  - An instance is "behind" if:
      there exist onboarding_instance_items where:
        instanceId = instance.id
        AND isRequired = true
        AND isCompleted = false
        AND dayNumber <= (currentDay - threshold)
  - Collect all behind instances with their hire's staffId, 
    name, teamId, and list of overdue day numbers

STEP 3 — Deduplicate — do not spam managers
  - For each behind instance, check if a "behind schedule" 
    notification was already sent for this instance TODAY
  - Search notifications for userId = managerId AND title 
    containing hire's name AND createdAt = today's date
  - If already sent today: skip this instance

STEP 4 — Find the manager for each behind hire
  - Use the same manager-lookup pattern from 
    onboardingLmsSyncService.ts (Prompt 6)
  - If no manager found: log warning and skip

STEP 5 — Send notification to the manager
  - title: "[First Last] is behind on onboarding"
  - message: "[First Last] has incomplete required items from 
    Day {earliest overdue day}. They are currently on Day 
    {currentDay} of {totalDays}."
  - userId: managerId

  Wrap all per-instance logic in try/catch — one failing 
  instance must not stop others.
